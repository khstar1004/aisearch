# 해오름기프트 Marqo-AI 기반 AI 상품 검색 최종 개발 계획

## 1. 프로젝트 목표

해오름기프트 기존 쇼핑몰에 **텍스트·이미지 통합 AI 상품 검색 기능**을 구축한다.

사용자는 다음 방식으로 상품을 검색할 수 있다.

```
1. 텍스트 검색
예: 검은 우산, 스텐 텀블러, 점착 메모지

2. 이미지 검색
예: 사용자가 상품 사진을 업로드하면 유사 상품 검색

3. 텍스트 + 이미지 혼합 검색
예: 텀블러 이미지 + “검은색”
예: 우산 이미지 + “3단 자동”
```

검색 결과는 세종기프트 AI 검색 기능을 참고하여 다음 형태로 제공한다.

```
- 상위 유사 상품 3개
- 비슷한 카테고리 추천
- 관련 상품 리스트
- 유사도 점수
- 상품번호, 상품명, 가격, 이미지, 상세 URL
```

## 2. 클라이언트 현재 환경 분석

현재 확인된 해오름기프트 환경은 다음과 같다.

| 항목 | 내용 |
| --- | --- |
| 기존 웹사이트 | 자체 서버 |
| DB | MSSQL |
| 상품 이미지 | 별도 이미지 서버 |
| 가맹점 구조 | 약 1,700개 사이트가 동일 DB 공유 |
| 신규 프로그램 설치 | 가능 |
| 설치 대상 | 리눅스 서버 |
| 서버 상태 | 2016년 구축 서버로 추정 |
| 요구 트래픽 | 최대 동시접속자 850명 수준 대응 필요 |

이 환경에서는 기존 쇼핑몰 코드를 직접 크게 수정하지 않고, **별도 AI 검색 서버를 구축하여 기존 사이트가 API 또는 JS 위젯으로 호출하는 방식**이 가장 적합하다.

## 3. 기술 선택: Marqo-AI

본 프로젝트의 AI 검색엔진 후보는 **Marqo-AI**를 우선 검토한다.

Marqo는 텍스트와 이미지 같은 멀티모달 데이터를 저장하고 검색할 수 있는 오픈소스 벡터 검색엔진으로 설명되며, 벡터 생성도 Marqo가 처리하는 구조다. 즉 Qdrant처럼 별도 임베딩 서버와 벡터DB를 직접 조립하는 방식보다 초기 개발이 단순하다.

Marqo는 로컬 이미지 검색 데모에서 텍스트·이미지 기반 멀티모달 검색과 weighted query를 활용한 이미지 검색 예제를 제공한다.

다만 중요한 리스크가 있다. Marqo GitHub에는 **Marqo Open Source project가 deprecated 되었고 더 이상 업데이트를 받지 않는다는 공지**가 명시되어 있다. 따라서 운영 시스템은 Marqo에 완전히 종속되지 않도록 **검색엔진 교체 가능한 구조**로 설계해야 한다.

## 4. 최종 권장 아키텍처

## 4.1 전체 구조

```
[해오름기프트 기존 웹사이트 / 1,700개 가맹점 사이트]
        ↓
[AI 검색 버튼 / JS 위젯 / 기존 검색창 연동]
        ↓ HTTPS
[Nginx / Load Balancer]
        ↓
[FastAPI AI 검색 API 서버 2대 이상]
        ↓
[Marqo 검색엔진 서버]
        ↑
[상품 동기화 Worker]
        ↑
[MSSQL 상품 DB - read-only]
        ↑
[별도 이미지 서버]
```

---

## 4.2 핵심 설계 원칙

```
1. 기존 MSSQL DB는 원본으로 유지한다.
2. 기존 DB에는 쓰기 작업을 하지 않는다.
3. AI 검색용 인덱스는 Marqo에 별도로 구축한다.
4. 기존 웹사이트는 AI 검색 API 또는 JS 위젯만 호출한다.
5. 동시접속자 850명 대응을 위해 API 서버는 수평 확장 가능하게 만든다.
6. Marqo 장애 또는 성능 이슈에 대비해 검색엔진 추상화 구조를 둔다.
```

---

## 5. 동시접속자 850명 대응 전략

동시접속자 850명은 “850명이 동시에 사이트에 접속해 있는 상태”를 의미할 가능성이 높다. 이 중 실제로 동시에 AI 검색을 실행하는 사용자는 일부일 가능성이 크다.

따라서 시스템은 다음 기준으로 설계한다.

| 구분 | 기준 |
| --- | --- |
| 동시접속자 | 최대 850명 |
| 동시 AI 검색 요청 | 30~100건 burst 대응 목표 |
| 일반 텍스트 검색 응답 | 1~2초 내외 목표 |
| 이미지 검색 응답 | 2~5초 내외 목표 |
| 부하 테스트 기준 | 100 concurrent search 요청 기준 검증 |
| 확장 방식 | API 서버 수평 확장 + 검색엔진 서버 증설 가능 구조 |

이미지 검색은 텍스트 검색보다 무겁다. 사용자가 이미지를 업로드하면 이미지 전처리와 임베딩 생성이 필요하기 때문이다. 따라서 이미지 검색 요청에는 **rate limit, 큐 처리, 캐싱, 파일 용량 제한**을 반드시 적용한다.

## 6. 서버 구성 계획

## 6.1 PoC 서버

초기 검증용.

```
CPU: 4 vCPU 이상
RAM: 8GB 이상
Disk: SSD 50GB 이상
GPU: 필수 아님
Docker / Docker Compose 필요
```

목적:

```
- 상품 300~500개 색인
- 텍스트 검색 테스트
- 이미지 검색 테스트
- 텍스트+이미지 혼합 검색 테스트
```

---

## 6.2 운영 MVP 서버

동시접속자 850명 대응을 고려한 최소 운영 구성.

```
[서버 1] API 서버
CPU: 4~8 vCPU
RAM: 8~16GB
역할: FastAPI, Nginx, JS 위젯 제공

[서버 2] Marqo 검색엔진 서버
CPU: 8 vCPU 이상
RAM: 32GB 권장
Disk: SSD 200GB 이상
역할: AI 검색 인덱스, 임베딩, 검색 처리

[서버 3] Sync Worker / Batch 서버
CPU: 4 vCPU 이상
RAM: 8GB 이상
역할: MSSQL 동기화, 이미지 다운로드, 재색인
```

예산이나 서버 여건이 부족하면 초기에는 API 서버와 Sync Worker를 한 서버에 같이 둘 수 있다.

다만 **Marqo 검색엔진 서버는 분리 권장**이다.

---

## 6.3 안정 운영 구성

동시 검색량이 많아질 경우 권장 구성.

```
[Nginx / Load Balancer]
        ↓
[FastAPI API 서버 2대]
        ↓
[Marqo 검색 서버 1대 이상]
        ↓
[Redis Cache]
        ↓
[MSSQL read-only]
```

추가 구성:

```
- Redis: 검색 결과 캐싱, rate limit
- Queue: 이미지 검색 요청 폭주 시 비동기 처리
- Monitoring: Prometheus/Grafana 또는 서버 로그 모니터링
- Log Storage: 검색 로그, 오류 로그, 색인 실패 로그 저장
```

## 7. GPU 사용 계획

1차 MVP에서는 GPU 없이 CPU 기반으로 시작한다.

다만 이미지 검색 요청이 많거나, 상품 수가 많거나, 대형 멀티모달 모델을 적용할 경우 GPU 서버를 추가 검토한다. Marqo 관련 자료에서도 GPU 사용 시 검색 또는 임베딩 처리 속도를 높일 수 있으며, CUDA 기반 사용이 언급된다.

## GPU 도입 기준

| 상황 | GPU 필요성 |
| --- | --- |
| 상품 300~500개 PoC | 불필요 |
| 상품 수천~수만 개 MVP | 없어도 가능 |
| 이미지 검색 요청이 많음 | 권장 |
| 대형 모델 사용 | 권장 |
| 전체 상품 재색인을 빠르게 해야 함 | 권장 |
| 응답속도 1~2초 이내 강제 | 권장 |

초기에는 CPU 서버로 시작하되, 운영 후 이미지 검색량과 응답속도 데이터를 보고 GPU 도입을 판단한다.

## 8. 데이터 연동 계획

## 8.1 MSSQL 연동

기존 MSSQL DB는 read-only로 접근한다.

기존 개발자에게 AI 검색용 View 생성을 요청한다.

필수 필드:

```
product_id 또는 p_idx
product_name
price
category_name
main_image_url
product_url
status
updated_at
is_deleted 또는 display_yn
mall_id 또는 site_id, 필요한 경우
```

예시:

```
SELECT
    product_id,
    product_name,
    price,
    category_name,
    main_image_url,
    product_url,
    status,
    updated_at,
    is_deleted
FROM dbo.v_ai_search_products
WHERE display_yn='Y';
```

---

## 8.2 이미지 서버 연동

상품 대표 이미지는 별도 이미지 서버에서 다운로드한다.

요구사항:

```
- AI 검색 서버에서 이미지 URL 접근 가능해야 함
- 이미지 서버 IP 제한 여부 확인
- 대량 다운로드 제한 여부 확인
- 대표 이미지와 상세 이미지 구분 필요
- 손상 이미지, 저화질 이미지, 너무 작은 이미지는 로그 처리
```

이미지 다운로드는 서버 부하를 막기 위해 제한한다.

```
- 동시 다운로드 3~5개부터 시작
- 야간 전체 색인 권장
- 실패 이미지 재시도
- 실패 로그 저장
```

## 9. Marqo 인덱스 설계

Marqo에는 상품 1개를 하나의 document로 저장한다.

예시 document:

```
{
  "_id":"P001",
  "product_id":"P001",
  "product_name":"3단 자동 우산",
  "category_name":"우산",
  "description":"판촉물 기프트 3단 자동 우산 접이식 우산",
  "price":8500,
  "main_image_url":"https://image-server.example.com/p001.jpg",
  "product_url":"https://mall.example.com/product_view.asp?p_idx=P001",
  "status":"active",
  "updated_at":1710000000
}
```

검색 대상 필드:

```
product_name
category_name
description
main_image_url
```

결과 출력/필터용 필드:

```
product_id
price
status
product_url
category_name
updated_at
mall_id
```

## 10. 검색 기능 개발 계획

## 10.1 텍스트 검색

사용자가 입력한 검색어를 기반으로 상품을 검색한다.

예시:

```
검은 우산
스텐 텀블러
점착 메모지
크리스탈 상패
고급 볼펜
```

결과는 다음 기준으로 정렬한다.

```
- AI 의미 유사도
- 상품명/카테고리 일치도
- 판매중 여부
- 필요 시 인기/추천 점수
```

---

## 10.2 이미지 검색

사용자가 이미지를 업로드하면 유사 상품을 검색한다.

처리 흐름:

```
이미지 업로드
→ 파일 검증
→ 이미지 리사이즈/전처리
→ Marqo 이미지 검색
→ 유사 상품 결과 반환
```

제한 정책:

```
- JPG, PNG, WEBP 허용
- 최대 5MB 또는 10MB 제한
- 비이미지 파일 차단
- 이미지 검색 요청 rate limit 적용
```

---

## 10.3 텍스트 + 이미지 혼합 검색

본 프로젝트의 핵심 기능이다.

예시:

```
이미지: 텀블러 사진
텍스트: 검은색
→ 검은색 계열 텀블러 우선 노출
```

```
이미지: 우산 사진
텍스트: 3단 자동
→ 3단 자동 우산 우선 노출
```

검색 로직:

```
1. 이미지 기반 유사도 검색
2. 텍스트 기반 의미 검색
3. 결과 병합
4. 카테고리/상태 필터 적용
5. 최종 점수 계산
```

초기 가중치:

```
이미지만 검색: 이미지 100%
텍스트만 검색: 텍스트 100%
혼합 검색: 이미지 60% + 텍스트 40%
```

PoC 결과에 따라 조정한다.

## 12. 동기화 계획

## 12.1 최초 색인

```
MSSQL View 조회
→ 상품 데이터 추출
→ 이미지 URL 확인
→ Marqo document 생성
→ Marqo 인덱스에 추가
```

초기에는 전체 상품이 아니라 300~500개 상품으로 검증한다.

---

## 12.2 정기 동기화

| 구분 | 주기 | 내용 |
| --- | --- | --- |
| 변경 상품 동기화 | 1시간마다 | `updated_at` 기준 신규/수정 상품 반영 |
| 비노출/삭제 검증 | 매일 새벽 | 삭제/비노출/품절 상품 제외 |
| 전체 재색인 | 필요 시 | 모델 변경, 인덱스 장애, 품질 개선 시 |
| 수동 재색인 | 필요 시 | 특정 상품 오류 수정 |

---

## 12.3 삭제/비노출 처리

초기에는 완전 삭제보다 `status = inactive` 처리 후 검색 시 제외하는 방식을 권장한다.

```
삭제/비노출 상품
→ Marqo document status inactive
→ 검색 API에서 status active만 반환
```

## 13. 1,700개 가맹점 대응 계획

1,700개 가맹점이 동일 DB를 공유하므로 AI 검색 인덱스는 공통으로 구축한다.

다만 아래 항목은 가맹점별 차이가 있을 수 있다.

```
- 상품 상세 URL
- 상품 가격
- 상품 노출 여부
- 디자인
- 도메인
```

따라서 API에는 `mall_id`를 포함한다.

```
{
  "mall_id":"shop001",
```

## 15.2 캐싱 전략

| 대상 | 방식 |
| --- | --- |
| 자주 검색되는 텍스트 | Redis 캐싱 |
| 동일 이미지 검색 | 이미지 hash 기반 캐싱 |
| 카테고리 추천 | 검색 결과 기반 캐싱 |
| 상품 상세 URL | API 응답 단계 캐싱 |

이미지 hash 예시:

```
사용자 업로드 이미지
→ perceptual hash 또는 SHA256
→ 동일 이미지 검색 결과 캐시 반환
```

---

## 15.3 Rate Limit

이미지 검색은 서버 부하가 크므로 제한한다.

예시 정책:

```
IP당 초당 1~3회 이미지 검색 제한
mall_id별 분당 요청 제한
이미지 업로드 용량 제한
과도한 요청은 429 응답
```

---

## 15.4 부하 테스트

운영 전 반드시 부하 테스트를 진행한다.

테스트 시나리오:

```
1. 텍스트 검색 100 concurrent
2. 이미지 검색 30 concurrent
3. 혼합 검색 30 concurrent
4. 전체 혼합 트래픽 850 active user 시뮬레이션
5. API 서버 1대/2대 비교
6. Marqo CPU/RAM 사용량 측정
```

합격 기준:

```
- 텍스트 검색 평균 1~2초 내외
- 이미지 검색 평균 2~5초 내외
- 오류율 1% 이하
- 서버 CPU/RAM 한계 도달 전 확장 기준 확인
- API 서버 장애 시 재기동 가능
```

## 22. 최종 개발 전략

본 프로젝트는 **Marqo-AI 기반 텍스트·이미지 통합 상품 검색 시스템**을 별도 서버로 구축하고, 기존 해오름기프트 웹사이트와 1,700개 가맹점 사이트가 공통으로 사용할 수 있는 API/위젯 형태로 제공한다.

최종 전략은 다음과 같다.

```
1. 상품 300~500개로 PoC 진행
2. Marqo로 텍스트/이미지/혼합 검색 품질 검증
3. 검색 품질이 적합하면 MVP 개발 진행
4. MSSQL read-only View와 이미지 서버를 연동
5. Marqo 인덱스에 전체 상품 색인
6. FastAPI 검색 API와 JS 위젯 구축
7. 동시접속자 850명 기준 부하 테스트 수행
8. 운영 적용 후 로그 기반으로 검색 품질 고도화
```

단, Marqo 오픈소스 업데이트 중단 리스크가 있으므로 다음 구조를 반드시 적용한다.

```
SearchEngine Interface
 ├─ MarqoSearchEngine
 ├─ TypesenseSearchEngine, 예비
 └─ QdrantSearchEngine, 예비
```

이렇게 설계하면 초기에는 Marqo로 빠르게 고도화된 AI 검색을 구현하고, 운영 중 기술 리스크가 발생할 경우 Typesense 또는 Qdrant/OpenCLIP 구조로 전환할 수 있다.
